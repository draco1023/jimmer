DTO语言

1. 概念​

DTO语言是Jimmer提供的一个非常强大的功能，是一个和对象抓取器高度类似的概念，但在编译过程中位于更早的阶段。

DTO语言用于快速定义数据结构的形状，根据这些形状，DTO可以在编译时

  • 生成特定数据结构所对应的静态DTO类型

  • 生成DTO静态对象和Jimmer动态对象之间的相互转换逻辑

  • 生成该形状所定义的对象抓取器

这种方法的开发效率很高，为Jimmer量身设计，远快于使用mapstruct，是优先推荐的方式。

提示

即使某些特殊场景下，我们无法直接使用统一的ORM实体，不得不使用DTO对象，开发代价也仅仅是定义这些DTO的形状而已，剩下的所有工作 (DTO相关Java/Kotlin定义、和实体之间的彼此转化逻辑、和数据库的交互逻辑) 都是全自动的，完全感觉不到传统技术方案中因DTO爆炸带来的痛苦。

2. 创建文件​

DTO语言的代码体现为扩展名为dto的文件，一旦编译完成，这些文件就没有价值了。所以，这些文件不能放到src/main/resources中打包，而是放到src/main/dto目录中。

因此，第一件事，是在src/main下建立dto子目录。

危险

dto文件由Jimmer的Annotation Processor (Java) 或Kotlin (Kotlin) 编译。因此，有两点注意事项

  • 对于多工程结构的项目而言，dto文件需要定义在实体类型所在子项目中。

  • 如果正在使用诸如Intellij这样的IDE开发项目

    • 如果除了dto文件外还有其他Java/Kotlin文件被修改了，直接点击IDE中运行按钮可以导致dto文件的重新编译

    • 但是，如果除了dto文件外没有其他Java/Kotlin文件被修改，简单地点击IDE中运行按钮并不会导致dto文件被重新编译，除非显式地rebuild！

如果你有好办法可解决这个问题，欢迎反馈。

假设存在一个Jimmer实体类型，其完整的类型名为com.yourcompany.yourproject.Book，该类被@org.babyfish.jimmer.sql.Entity修饰 (DTO语言只支持Entity类型)，那么

  1. 在src/main/dto目录下建立目录com/yourcompany/yourproject，即，建立和包名一致的目录

  2. 在上一步建立的目录中新文件Book.dto，该文件必须和类同名，且扩展名为dto

用户可以编辑Book.dto文件，定义任意个以Book类型为聚合根的形状。这里，我们先定一个DTO形状：

Book.dto
BookView {
    ...略...
}

编译后会生成Java/Kotlin类型BookView

  • Java
  • Kotlin
BookView.java
package com.yourcompany.yourproject.dto;

import com.yourcompany.yourproject.Book;
import org.babyfish.jimmer.View;

@lombok.Data
public class BookView implements View<Book> {
    ...略...
}
BookView.kt
package com.yourcompany.yourproject.dto

import com.yourcompany.yourproject.Book
import org.babyfish.jimmer.View

data class BookView(
    ...略...
) : View<Book> {
    ...略...
}
信息

  • 生成的类型所在的包为com.yourcompany.yourproject.dto，相对实体所在的包多了一个dto子包。

  • 对于Java而言，其生产的类型假设用户使用了lombok。

3. input、input-only修饰符​

形状可以被input修饰符修饰，例如

Book.dto
BookView {
    ...略...
}

AnnotherBookView {
    ...略...
}

input BookInput {
    ...略...
}

input AnnotherBookInput {
    ...略...
}

这表示

  • BookView和AnnotherBookView用作查询输出，生成的Java/Kotlin类型会实现org.babyfish.jimmer.View&lt;E&gt;接口

    备注

    建议输出形状为View结尾

  • BookInput和AnnotherBookInput用作保存输入，生成的Java/Kotlin类型会实现org.babyfish.jimmer.Input&lt;E&gt;接口

    备注

    建议输入形状为Input结尾

  • 事实上，还有另外一种没有直接的Input/Output语义的DTO对象，往往用做复杂动态查询API的查询规格参数。 如例子中的AuthorSpecification类型定义。

    备注

    Jimmer对这种查询规则参数类型是否应该定义为input类型不做任何建议，但是建议这种类型以Specification/Spec结尾。

信息

和Output DTO相比，Input DTO存在如下不同

  • 如果实体id属性配置了自动增长策略，那么input DTO中的id属性是nullable的。

  • input DTO中只能定义可以保存的属性，如简单属性、普通ORM关联属性和id-view属性。 不能定义无法保存的属性，如transient属性、公式属性、计算属性和远程关联，否则会导致编译错误。

无论是输出形状还是输入形状，其生成的Java/Kotlin类型具备如下功能

  • 构造方法：将Jimmer动态实体对象转化为静态DTO对象

  • toEntity()：将静态DTO对象转化为Jimmer动态实体对象

以BookView为例

  • Java
  • Kotlin
Book entity = ...略...

// 实体 -> DTO
BookView dto = new BookView(entity);

// DTO -> 实体
Book anotherEntity = dto.toEntity();
val entity: Book = ...略...

// 实体 -> DTO
val dto = BookView(entity)

// DTO -> 实体
val anotherEntity: Book = dto.toEntity()

除了input外，还可以使用input-only或inputOnly修饰符修饰，例如

Book.dto
input-only BookUpdateInfo {
    ...略...
}

或

Book.dto
inputOnly BookUpdateInfo {
    ...略...
}

input-only/inputOnly和input的作用类似，用于修饰输入类型，但只提供从DTO对象转化为实体对象的功能，不提供从实体对象转化为DTO对象的功能。

input-only/inputOnly修饰符的用途，请参见7. 可空性

5. allScalars​

DTO语言支持一个特别的宏属性，#allScalars，表示映射实体中的所有标量属性

Book.dto
BookView {
    #allScalars
}

按照官方例子中Book类型的定义

  • Book本身的标量属性有id、name、edition和price

  • Book继承了TenantAware，TenantAware接口定义了属性tenant

  • Book也继承了CommonEntity (多继承)，CommonEntity接口定义了属性createdTime和modifiedTime

#allScalars表示自动映射实体所有标量属性，包括继承的。

如果这不是你所要的行为，那么可以使用allScalars(Book)，表示只映射Book类型本身定义的所有标量属性，不包含继承的属性。

事实上，#allScalars支持多个参数，举例如下

宏表达式自动映射的属性
#allScalarscreatedTime, modifiedTime, tenant, id, name, edition, price
#allScalars(Book)id, name, edition, price
#allScalars(TenantAware)tenant
#allScalars(CommonEntity)createdTime、modifiedTime
#allScalars(Book, TenantAware)tenant, id, name, edition, price
#allScalars(Book, CommonEntity)createdTime、modifiedTime, id, name, edition, price
#allScalars(TenantAware, CommonEntity)createdTime、modifiedTime, tenant
#allScalars(Book, TenantAware, CommonEntity)createdTime, modifiedTime, tenant, id, name, edition, price
信息

  • 如果使用#allScalars宏，则它必须被定义成第一个属性，否则，否则会导致编译报错。

  • 如果为#allScalars宏指定参数，则每个参数必须为当前实体或其基类型，否则会导致编译报错。

6. 负属性​

前文所讲的#allScalars宏，和后文会讲的继承，都会批量化地让DTO映射多个属性。负属性可以去掉某些属性

Book.dto
BookView {
    #allScalars
    -tenant
}

和上面一样，如果按官方例子中Book类型的继承关系来理解的话，#allScalars会加入属性createdTime、modifiedTime、tenant、id、name、edition和price。

而-teant从中减去了tenant属性，最终加入属性createdTime、modifiedTime、id、name、edition和price。

信息

负属性指定的其实不是实体属性名，而是DTO中对应属性的名称，在这里二者相同，并无差异。

稍后的章节重命名会揭示二者区别。

不难发现，对于这个例子而言，还有一种与之等价的写法

Book.dto
BookView {
    // 排除`TenantAware`
    #allScalars(Book, CommonEntity) 
}

7. 可空性​

默认情况下，DTO属性的可空性和原始属性的可空性一样。但是我们可以采用问号?或!修饰DTO属性，改变DTO属性的可空性。

7.1 ?​

我们可以采用?修饰DTO属性，让其可null。

Book.dto
input UpdateBookInput {
    #allScalars
    price?
}

在实体中Book.price不能为null，而对应的DTO属性UpdateBookInput.price可以为null，看似矛盾，但Jimmer可以处理这样的矛盾 (暂不讨论特定场景下使用该功能是否妥当)。

  • 把实体转化为DTO时，如果Book.price未加载 (虽然该属性不能为null，但动态对象可以拥有未加载属性)，那么转化后的UpdateBookInput.price为null。

  • 把DTO转化为实体时，如果UpdateBookInput.price为null，那么转化后的Book.price处于未加载状态，即，动态对象的这个属性根本不会被赋值。

你甚至可以对allScalars采用?，让所有自动映射的非关联属性全部可以为null

Book.dto
input UpdateBookInput {
    #allScalars?
}
警告

然而，此功能需要谨慎使用，静态类型的意义就应该固定和明确的，而非去追求如动态类型一般的灵活多变。

对于本身就可以为null的实体属性而言，具有三种状态: 非null、null和unloaded (未知)；而静态DTO属性只有两种状态：非null和null。即，引入静态DTO类型这件事本身就已经导致了信息丢失。

所以，对于本身就可以为null的实体属性而言，无论对应的DTO属性如何设置，经过DTO转化后的动态实体对象，其可空的属性一定为被赋值 (可能为null)，但不可能出现unloaded的情况。如果将其传递给保存指令，null一定会导致表中对应字段被update，而无法如同未被赋值的非null属性一样做到根本不更新。

对于没有直接Input/Output语义的查询规格参数Specification而言，比如AuthorSpecification， 这个功能是非常适合的，这是因为动态查询的每个查询参数都应该是可选的，除非因性能问题而不得不要求用户必须给出部分查询条件。

但是，对于其他具备Input/Output语义的DTO类型而言，不推荐使用此功能。

7.2 !​

我们可以采用!修饰DTO属性，让其非null。但其使用场景受限，只能用于以下两种情况。

  • 在input/input-only类型中修饰id属性

    前文提过，如果id属性被配置自动增长策略，那么对应的input DTO类型将会把id属性设置为可null。

    然而，这种行为并不总是符合用户预期，可以在input/input-only类型中使用!修饰id属性，让其非null，例如

    Book.dto
    input BookUpdateInfo {
        #allScalars
        id!
    }

  • 在input-only中将可null的实体属性转化为非null，例如

    Book.dto
    input-only BookUpdateInfo {
        #allScalars
        store! {
            ...略...
        }
    }
    信息

    实体中属性可以为null，但是DTO中对应属性非null，所以，无法支持将实体对象转化为DTO对象，只能把DTO对象转化为实体对象。

    这也是input-only或inputOnly关键字的含义。

8. 重命名​

可以使用as关键字为属性设置别名，让实体属性名和DTO属性名不同。

as有两种用法，既可以精确地为一个属性设置别名，也可以模糊地为多个属性设置别名

8.1. 重命名单个属性​

Book.dto
BookView {
    name as bookName
}

8.2. 重命名多个属性​

假设有一个实体类型叫做Robot

Robot.dto
OrderView {
    as (^ -> oldSystem) { ❶ 
        prop1
        prop2
    }
    as ($ -> ForNewSystem) { ❷
        prop3
        prop4
    }
}

其中

  • ❶ 为prop1和prop2添加前缀oldSystem，其中^表示起始位置

  • ❷ 为prop3和prop4添加后缀ForNewSystem，其中$表示起始位置

生成的RobotView类如下

  • Java
  • Kotlin
BookView.java
@lombok.Data
public class RobotView implements View<Robot> {

    private String oldSystemProp1;
    private String oldSystemProp2;

    private String prop3ForNewSystem;
    private String prop4ForNewSystem;

    ...省略其他成员...
}
BookView.kt
data class RobotView(
    
    val oldSystemProp1 = "",
    val oldSystemProp2 = "",

    val prop3ForNewSystem = "",
    val prop4ForNewSystem = ""
) : View<Robot> {

    ...省略其他成员...
}

可以在as() {...}块中定义绝大部分属性，当然包括#allScalars，例如

Robot.dto
RobotView {
    as(^ -> robot) {
        #allScalars
    }
}

下面罗列as() {...}块的所有用法

代码示范作用描述
as(^ -> prefix)添加前缀
as(^prefix ->)删除前缀
as(^prefix -> newPrefix)替换前缀
as($ -> suffix)添加后缀
as($suffix ->)删除后缀
as($suffix -> newSuffix)替换后缀
as(infix -> )删除任何位置的内容
as(infix -> newInfix)替换任何位置的内容
警告

^和$不能同时出现，因为这表示精确重命名单个属性，这和已有的功能重复

10. 关联属性​

对关联属性的处理是DTO语言一个非常强大的功能，存在三种用法

  • 直接使用
  • 递归关联
  • 调用id函数后使用
  • 调用flat函数后使用

10.1. 普通关联​

  • 基本用法

    Robot.dto
    input CompositeBookInput {

            -id

    生成的代码为

10.2 递归关联​

对于实体中可递归的属性，比如，附带例子中的TreeNode.parent或TreeNode.childNodes，可以使用星号*将DTO对应的关联属性标记为递归属性。

TreeNode.dto
TreeNodeView {
    
    #allScalars

    childNodes {
        #allScalars
    }*
}

10.3. id函数​

对于

而言，我们当然选择在实体中定义@IdView属性，例如

  • Java
  • Kotlin
Book.java
@Entity
public interface Book {

    @Nullable
    @ManyToOne
    BookStore store();

    @ManyToMany
    val authors: List<Author>

    @IdView
    Long storeId();

    @IdView("authors")
    List<Long> authorIds();
}
Book.kt
@Entity
interface Book {

    @ManyToOne
    val store: BookStore?

    @ManyToMany
    val authors: List<Author>
    
    @IdView
    val storeId: Long?

    @IdView("authors")
    val authorIds: List<Long>
}

在实体已经定义了@IdView属性的情况下，只需要用DTO语言简单地引用即可

Book.dto
BookView {

    storeId

    authorIds

    ...略...
}
提示

然而，我们不能总是寄希望于实体类型总是声明了@IdView属性，更好的方案不应该有此假设。

因此，DTO语言支持对关联属性调用id函数来达到同样目的

Book.dto
BookView {

    id(store)

    id(authors) as authorIds

    ...略...
}

这段代码的功能和前面的例子完全一样，但不再假设实体类型总是声明了@IdView属性，是更好的方案。

10.4. flat函数​

flat函数只能用于引用关联 (一对一或多对一)，不能用于集合关联 (一对多或多对多)，用于把关联DTO的属性提升到当前DTO类型，得到不含关联的平坦对象。

以官方例子中的TreeNode实体为例 (事实上，这个例子是错误的，我们稍后会讨论这个问题)

TreeNode.dto
FlatTreeNodeView {
    
    #allScalars(TreeNode)

    flat(parent) {
        #allScalars(TreeNode)
    }
}

如上文所述，这个例子是错误的，因为聚合根和关联对象都有属性id和name，无条件把关联对象的属性往聚合根上提取，必然导致名字冲突，最终导致DTO语言编译报错。

正确的做法，是和用户多属性模糊重命名的as(...) {}块结合使用。让我们来看一个新的例子

TreeNode.dto
FlatTreeNodeView {
    
    #allScalars(TreeNode) // id, name

    flat(parent) {
        as(^ -> parent) {
            // parentId, parentName
            #allScalars(TreeNode) 
        }
    }
}

这样，我们得到了一个平坦的DTO类型，具有4个属性：id、name、parentId和parentName。

        as(^ -> parent) {
            #allScalars(TreeNode)
        }
        flat(grandParent) {
            as(^ -> parent) {
                #allScalars(TreeNode)
            }
            
        }   
    }
}

这样，我们得到了一个平坦的DTO类型，具有6个属性：id、name、parentId、parentName、grandParentId、grandParentName。

11. 继承​

DTO类型定义之间支持继承，具备如下特色

  • DTO类型定义之间的继承，只是配置层面的继承而已。因此，既和原始实体的继承关系无关，也和生成的DTO的类型无关 (截止到目前版本为止，不会生成带继承关系的Java/Kotlin类)。

  • DTO类型之间的继承只针对dto文件内部。

  • DTO类型之间的继承是多继承，即，一个DTO类型定义，可以从多个DTO类型继承。

  • input类型定义只能继承自input类型定义。

例子如下:

Robot.dto

BaseRobotView1 {
    ...略...
}

BaseRobotView2 {
    ...略...
}

DefaultRobotView : BaseRobotView1, BaseRobotView2 {
    ...略...
}

对于生成的Java/Kotlin类型DefaultRobotView类而言，并不会有任何Java/Kotlin继承关系，这是因为：

  • 生成的是Java/Kotlin类而非接口，无法实现多继承

  • 由于负属性特性的存在，相比于基DTO类型定义而言，派生DTO类型定义可能反而少了一些属性

这就是"DTO类型定义之间的继承，只是配置层面的继承而已"的根本原因。

但是，上述代码，却会导致在Java/Kotlin中生成BaseRobotView1类和BaseRobotView2类。

如果你期望基DTO类型定义只为派生DTO类型定义所用，其本身不生成任何Java/Kotlin类型，可以添加abstract修饰符，如下

Robot.dto

abstract BaseRobotView1 {
    ...略...
}

abstract BaseRobotView2 {
    ...略...
}

DefaultRobotView : BaseRobotView1, BaseRobotView2 {
    ...略...
}

12. 自定义字段​

12.1 初识自定义字段​

前面我们所讲的例子中，所有DTO属性都是由实体属性映射而来，成为映射属性。

除了映射属性外，DTO类型还支持映射属性

Author.dto
import java.util.LocaleDateTime ❶

AuthorSpecification {
    
    #allScalars(Author)? - id

    minCreatedTime: LocaleDateTime?  ❷
    maxCreatedTimeExclusive: LocaleDateTime?  ❸
}

  • ❶ 导入类型，简化后续自定义属性的定义

  • ❷ ❸ 定义自定义属性的定义

    自定义属性采用类似于kotlin的语法，可以采用?明确定属性可为null，否则，不能为null。

    • 对于Kotlin项目而言，生成与之完全对应的类型

    • 对于Java项目而言，除了顶层类型的?会被用于在目标Java字段上生成@org.jetbrains.annotations.Nullable注解外，内部范型的?会被忽略

    警告

    这里的自定义的属性允许为null，如果要定义非null的属性 (即去掉?)，必须使用input-only/inputOnly修饰符

这里，AuthorSpecification并没有直接的Input/Output语言，被用作复杂动态查询的查询规格参数，例如

  • Java
  • Kotlin
AuthorRepository.java
public interface AuthorRepository extends JRepository<Author, Long> {
    
    AuthorTable table = AuthorTable.$;

    default List<Author> find(AuthorSpecification spec) {
        return sql()
            .createQuery(table)
            .whereIf(
                StringUtils.hasContent(spec.getFirstName()),
                table.firstName().ilike(spec.getFirstName() != null)
            )
            .whereIf(
                StringUtils.hasContent(spec.getLastName()),
                table.lastName().ilike(spec.getLastName() != null)
            )
            .whereIf(
                spec.getGender() != null,
                table.gender().eq(sepc.getGender());
            )
            .whereIf(
                spec.getMinCreatedTime() != null,
                () -> table.createdTime().ge(spec.getMinCreatedTime())
            )
            .whereIf(
                spec.getMaxCreatedTimeExclusive() != null,
                () -> table.createdTime().lt(spec.getMaxCreatedTimeExclusive())
            )
            .select(table)
            .execute()
    }
}
AuthorRepository.kt
interface AuthorRepository : KRepository<Author, Long> {

    fun find(spec: AuthorSpecification): List<Author> =
        sql
            .createQuery(Author::class) {
                spec.firstName?.takeIf { it.isNotEmpty() }?.let {
                    where(table.firstName ilike it)
                }
                spec.lastName?.takeIf { it.isNotEmpty() }?.let {
                    where(table.lastName ilike it)
                }
                spec.gender?.let {
                    where(table.gender eq it)
                }
                spec.minCreatedTime?.let {
                    where(table.createdTime ge it)
                }
                spec.maxCreatedTimeExclusive?.let {
                    where(table.createdTime lt it)
                }
                select(table)
            }
            .execute()
}

对于其他非关联字段而言，查询条件是eq和ilike这样的单值查询条件，所以简单地使用#allScalars映射即可。

然而，createdTime所对应的查询条件相对复杂，是一个区域查询，并不能简单地对映射实体中的createdTime属性，而是需要两个边界属性：minCreatedTime和maxCreatedTimeExclusive。

由此可见，对于AuthorSpecification这种查询规格参数而言，自定义属性能带来极大的便利，能映射的属性就映射，不能映射的属性就自定义。

因此，建议如下

  • Specificiation DTO：鼓励使用，自定义属性几乎就是为这种类型设计。

  • Input DTO: 可以少量使用，允许在Input DTO中混杂少量控制信息。

    DTO对象的toEntity方法会忽略这些信息，把其他信息转化为一个干净的动态实体对象供保存指令使用

  • Output DTO: 不鼓励使用

    • 首先，Output DTO本身不是必须的，让查询方法直接返回动态对象，并辅以@FetchBy注解，已经是足够强大的方案了。

    • 其次，即便使用了Output DTO，Output DTO对象也只会拥有实体对象的属性，除非你愿意手动填充自定义属性。

    提示

    对于输出类型而言，自定义属性是实体层面的事 (所以Output DTO不是必须的)。

    请参见*视图属性、简单计算属性、或复杂计算属性

12.4 导入语句​

除了内置类型外的其他类型外，其他类型在被引用时，要么书写全名，要么在文件开头使用import语句，否则，将会认为其同当前实体属于同一包。

DTO语言支持多种风格的import语句，例如

  • 单类导入

    import java.time.LocalDateTime

  • 单类重命名导入

    import java.time.LocalDateTime as LDT

  • 多类导入

    import java.time.{ 
        LocalDateTime, 
        OffsetDataTime, 
        ZonedDataTime 
    }

  • 多类重命名导入

    import java.time.{ 
        LocalDateTime as LDT, 
        OffsetDataTime as ODT, 
        ZonedDataTime as ZDT
    }

13. 注解​

13.1 基本用法​

你可以在DTO语言中使用注解修饰DTO类型、映射属性和自定义属性

Author.dto
import org.babyfish.jimmer.client.Doc
import org.springframework.format.annotation.DateTimeFormat

@Doc("Author对象复杂查询的查询规格") ❶
AuthorSpecification {
    
    #allScalars

    @Doc("模糊匹配")  ❷
    firstName

    @Doc("模糊匹配")  ❸
    lastName

    @Doc("最小创建时间，闭") ❹
    @DateTimeFormat(pattern = "yyyy-MM-dd") ❺
    minCreatedTime: LocalDateTime?

    @Doc("最小创建时间，开") ❻
    @DateTimeFormat(pattern = "yyyy-MM-dd") ❼
    maxCreatedTimeExclusive: LocalDateTime?
}

其中

  • ❶ 修饰类型

  • ❷ ❸ 修饰映射属性

  • ❹ ❺ ❻ ❼ 修饰自定义属性

提示

有一个细节需要注意

BookView
...省略import...

    ...省略其他属性...

    @A authors @B {
        ...省略关联对象属性...
    }

    ...省略其他DTO类型...
}

其中

  • @A修饰authors属性

  • @B修饰匿名的关联类型

13.2 注解替换​

  • 如前文所述，DTO语言支持注解，直接控制被生成的DTO类型的注解

  • 如果DTO属性没有被任何注解修饰，就会复制原实体中对应属性的注解 (如果DTO属性是映射而来，而非自定义的)。 其中，除org.babyfish.jimmer.client包下的其他任何 非 jimmer注解都会被复制到被生成的DTO类型中

12.3 value参数​

注解中的value参数可以被简写，例如

@UserAnnotation(value = "a", value2 = "b", value3 = "c")

可以被简写为

@UserAnnotation("a", value2 = "b", value3 = "c")

和Java不同，无论注解有多少个参数，value参数都可以被缩写，只要保证被缩写的属性最先配置即可。

13.4 混合Java和Kotlin的语法​

Java和Kotlin的注解语法存在一些微弱的差异，DTO语言混合了二者的特征，例如

  • @UserAnnotation(
        "key1",
        items = {
            @Item("subKey1", description = "Detail information for subKey1 ..."),
            @Item("subKey2", description = "Detail information for subKey2 ..."),
            @Item("subKey3", description = "Detail information for subKey3 ...")
        }
    )
  • @UserAnnotation(
        "key1",
        items = [
            @Item("subKey1", description = "Detail information for subKey1 ..."),
            @Item("subKey2", description = "Detail information for subKey2 ..."),
            @Item("subKey3", description = "Detail information for subKey3 ...")
        ]
    )
  • @UserAnnotation(
        "key1",
        items = {
            Item("subKey1", description = "Detail information or subKey1 ..."),
            Item("subKey2", description = "Detail information for subKey2 ..."),
            Item("subKey3", description = "Detail information for subKey3 ...")
        }
    )
  • @UserAnnotation(
        "key1",
        items = [
            Item("subKey1", description = "Detail information for subKey1 ..."),
            Item("subKey2", description = "Detail information for subKey2 ..."),
            Item("subKey3", description = "Detail information for subKey3 ...")
        ]
    )

以上四种写法是等价的。你可以随意选择你喜欢的系法。
