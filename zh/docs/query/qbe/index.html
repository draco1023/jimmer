QBE查询

QBE查询，就是用一个现有的实体对象作为模板，进行动态查询。

这里的模板实体对象是一个孤单对象，除了基于外键的关联对象属性可以设置为null或只有id属性的关联对象外，不得具备其他任何关联。

基本用法​

  • Java
  • Kotlin
public void findBooks(Book template) {
    return sqlClient.getEntities().findByExample(
        Example.of(template)
    );
}
fun findBooks(template: Book): List<Book> =
    return sqlClient.entities.findByExample(
        example(template)
    )

  • 例1：仅指定name

    • Java
    • Kotlin
    List<Book> books = findBooks(
        BookDraft.$.produce(draft -> {
            draft.setName("GraphQL in Action");
        })
    );
    val books = findBooks(
        new(Book::class).by {
            name = "GraphQL in Action"
        }
    )

    得到的SQL如下

    select 
        tb_1_.ID, 
        tb_1_.NAME, 
        tb_1_.EDITION, 
        tb_1_.PRICE, 
        tb_1_.STORE_ID
    from BOOK tb_1_
    where
        tb_1_.NAME = ? /* GraphQL in Action */

  • 例2：仅指定edition

    • Java
    • Kotlin
    List<Book> books = findBooks(
        BookDraft.$.produce(draft -> {
            draft.setEdition(3);
        })
    );
    val books = findBooks(
        new(Book::class).by {
            edition = 3
        }
    )

    得到的SQL如下

    select 
        tb_1_.ID, 
        tb_1_.NAME, 
        tb_1_.EDITION, 
        tb_1_.PRICE, 
        tb_1_.STORE_ID
    from BOOK tb_1_
    where
        tb_1_.EDITION = ? /* 3 */

  • 例3：同时指定name和edition

    • Java
    • Kotlin
    List<Book> books = findBooks(
        BookDraft.$.produce(draft -> {
            draft.setName("GraphQL in Action");
            draft.setEdition(3);
        })
    );
    val books = findBooks(
        new(Book::class).by {
            name = "GraphQL in Action"
            edition = 3
        }
    )

    得到的SQL如下

    select 
        tb_1_.ID, 
        tb_1_.NAME, 
        tb_1_.EDITION, 
        tb_1_.PRICE, 
        tb_1_.STORE_ID
    from BOOK tb_1_
    where
            tb_1_.NAME = ? /* GraphQL in Action */ 
        and
            tb_1_.EDITION = ? /* 3 */
信息

不难看出，QBE是对简单动态查询的简化形式。

和JPA/Hibernate不同，Jimmer的QBE查询依赖于Jimmer实体对象特有的动态性，不依赖于Java语言不区分可空类型和非空类型的“错误”特性。

精细控制​

或许你已经发现，findByExample的参数并非模版实体对象，而是一个Example对象 (Java下为Example<E>, Kotlin下为KExample<E>)。

Example对象是模版实体对象的包装，提供API控制QBE查询精确行为。

Trim​

Trim开关默认关闭，开发人员可以开启它，如果模板对象的某个字符串属性被指定为非空值时，先进行trim再用做查询条件。

  • Java
  • Kotlin
List<Book> books = sqlClient.getEntities().findByExample(
    Example
        .of(
            BookDraft.$.produce(draft -> {
                draft.setName("         GraphQL in Action      ");
            })
        )
        .trim()
);
val books = sqlClient.entities.findByExample(
    example(
        new(Book::class).by {
            name = "         GraphQL in Action      "
        }
    ) {
        trim()
    }
)

上面的例子，用户指定的参数为" GraphQL in Action "，包含大量空白。

然而，我们调用了trim()方法，Jimmer先将该字符串进行trim，然后在进行查询，最终生成的SQL为

select 
    tb_1_.ID, 
    tb_1_.NAME, 
    tb_1_.EDITION, 
    tb_1_.PRICE, 
    tb_1_.STORE_ID
from BOOK tb_1_
where
    // 去除了空白
    tb_1_.NAME = ? /* GraphQL in Action */

上面代码中的trim()会对模板对象的任何被指定为非空的字符串属性进行处理，你也可以trim特定字符串属性，如下：

  • Java
  • Kotlin
Example
    .of(template)
    .trim(BookProps.NAME)
example(example) {
    trim(Book::name)
}

MatchMode​

Jimmer为QBE提供匹配模式选项: MatchMode，该选项是一个枚举

  • Java
  • Kotlin
package org.babyfish.jimmer.sql.ast.query;

public interface Example<E> {

    ...省略其他成员...

    enum MatchMode {
        NOT_EMPTY,
        NOT_NULL,
        NULLABLE
    }
}
package org.babyfish.jimmer.sql.kt.ast.query

class Example<E: Any> {

    ...省略其他成员...

    enum class MatchMode {
        NOT_EMPTY,
        NOT_NULL,
        NULLABLE
    }    
}
是否添加where条件NULLABLENOT_NULLNOT_EMPTY(默认)
未指定的属性否否否
被指定为null的属性是(is null)否否
被指定为non-null的非字符串属性是是是
""是是否
长度不为0的字符串是是是
警告

Jimmer的QBE查询严格区分模板实体对象中的“未指定的属性”和“被指定为null的属性”，这是Jimmer实体的动态性特有的能力。

然而，如果你的模板实体对象是从DTO对象(其实就是静态POJO对象)转化而来，将会因信息丢失导致无法区分二者，请小心此问题(这也是MatchMode默认值并非NULLABLE的原因)。

  • 例1：默认的NOT_EMPTY或NOT_NULL

    • Java
    • Kotlin
    List<Book> books = sqlClient.getEntities().findByExample(
        Example
            .of(
                BookDraft.$.produce(draft -> {
                    draft.setStore(null)
                })
            )
    );
    val books = sqlClient.entities.findByExample(
        example(
            new(Book::class).by {
                store = null
            }
        )
    )

    这里，虽然Book.store指定了成了null，但是，默认的MatchMode会忽略它。所以，不会生成任何where条件，SQL如下

    select 
        tb_1_.ID, 
        tb_1_.NAME, 
        tb_1_.EDITION, 
        tb_1_.PRICE, 
        tb_1_.STORE_ID
    from BOOK tb_1_

  • 例2：MatchMode.NULLABLE

    • Java
    • Kotlin
    List<Book> books = sqlClient.getEntities().findByExample(
        Example
            .of(
                BookDraft.$.produce(draft -> {
                    draft.setStore(null)
                })
            )
            .match(Example.MatchMode.NULLABLE)
    );
    val books = sqlClient.entities.findByExample(
        example(
            new(Book::class).by {
                store = null
            }
        ) {
            match(KExample.MatchMode.NULLABLE)
        }
    )

    这里，虽然Book.store指定了成了null，同时，MatchMode被设置成了NULLABLE。所以，会生成where条件，SQL如下

    select 
        tb_1_.ID, 
        tb_1_.NAME, 
        tb_1_.EDITION, 
        tb_1_.PRICE, 
        tb_1_.STORE_ID
    from BOOK tb_1_
    where
        tb_1_.STORE_ID is null

上面代码中的match(...)会对模板对象的任何被指定了的属性形成影响，你也可以只影响特定属性，如下：

  • Java
  • Kotlin
Example
    .of(template)
    .trim(BookProps.NAME)
example(example) {
    trim(Book::name)
}
